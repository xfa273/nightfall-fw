# Nightfall FW 開発運用方針（暫定）

このドキュメントは、`nightfall-fw` の開発運用方針をまとめたものです。
今シーズンは `STM32F405` 系既存機体と `STM32F413` 系新機体を並行して開発し、**MCU系列ごとに1個の共通バイナリ**を運用する前提で更新します。

---

## 1. 目標

- `STM32F405` 系と `STM32F413` 系を同一リポジトリで並行開発できる構造にする
- 生成物（ELF/BIN）は **MCU系列ごとに1個**とし、機体差分は起動時の機体識別で切り替える
- ハード依存実装を `platform` / `board` に分離し、探索・経路導出・制御ロジックは可能な限り共通化する
- `STM32F405` 系は現行機体で完了とし、`F405 + FRAM` のような新規派生は作らない
- 公式にサポートする MCU系列の生成物は常にビルドできる状態を維持し、壊れ検知を早くする

---

## 2. リポジトリと命名

- メインリポジトリ名: `nightfall-fw`
- MCU系列名:
  - `stm32f405`
  - `stm32f413`
- ファミリ名:
  - `mini`
  - `classic`
- 正式な機体識別名:
  - `<family>_r<major>_<minor>`
  - 例: `mini_r1_1`
  - 例: `classic_r1_0`
- 命名の意味:
  - `major`: メカを含む機体系世代（基本性能や特性が大きく変わる単位）
  - `minor`: 同一機体系内の基板版（パターン修正、ピンアサイン変更、IC変更など）
- 作業名/仮称:
  - 例: `f413_preorder`
  - `_sandbox/` や実験ブランチでは許容するが、`main` に入れる前に正式名へ置き換える
- 既存の `mini_v1_1`, `classic_v2` は移行期間中の互換名として扱い、新規追加では使わない
- CMakeターゲット名（例）:
  - `nightfall_stm32f405`
  - `nightfall_stm32f413`
- 機体個体識別子（`board_id` / `unit_serial`）は **ビルドターゲット名とは分けて管理**し、起動時に機体識別ブロックから読み出す

## 3. ディレクトリ構成（推奨）

- `common/`
  - 迷路探索、経路導出、制御ロジック、共通ユーティリティ
- `app/`
  - アプリ層（状態機械、モード遷移、共通ユースケース）
- `platform/<mcu_family>/`
  - CubeMX生成物、HAL、startup、リンカスクリプト、MCU依存初期化
- `board/<board_name>/`
  - ピン割り当て、搭載デバイス有無、ボード初期化、機体能力テーブル
- `params/<profile>/`
  - 走行・調整パラメータ
- `nvm/`
  - 内蔵Flash / 外付けFRAM を抽象化する永続化層
- `stable/<family>/<unit_serial>/`
  - 個体ごとの安定バージョン定義、運用設定、メモ
- `tools/flashing/`
  - 書き込み・NVM初期化ツール
- `tools/logging/`
  - ログ吸い出し、変換、可視化ツール
- `docs/`
  - 運用ドキュメント、計画、実験記録、データシート
- `hardware/<board_name>/`
  - `cad/`, `bom/`, `notes/` などのハードウェア関連データ

### 3.1 CubeMX設定変更のルール

- `platform/<mcu_family>/cubemx_project/<project_name>/` 配下の `*.ioc` を正とする
- CubeMXで設定可能な項目（ピン割り当て、周辺機能有効化、Clock、NVIC、GPIO初期設定など）は、**生成コードを直接編集せず**、`*.ioc` を更新して反映する
- 変更は以下の順序で行う
  1. `*.ioc` を更新
  2. CubeMXコード再生成
  3. 差分確認
  4. 対象 MCU系列のビルド確認
  5. 共通ファイルに影響する場合は他の公式サポート系列もビルド確認
- `Core/Src` や `Core/Inc` のうち、CubeMXが再生成する領域は手編集しない
- 手編集が必要な実装は、原則として `/* USER CODE BEGIN */ ... /* USER CODE END */` ブロック、または `common/` / `app/` / `board/` / `params/` / `nvm/` 側に置く

### 3.2 正式データ置き場のルール

- 部品データシートは `docs/datasheets/` に置く
- CAD原本、BOM、ピン一覧、回路メモは `hardware/<board_name>/` にまとめる
- `_sandbox/` は一時検証用とし、参照元として定着させない
- マージ前に `_sandbox/` の成果物は正式ディレクトリへ移すか、不要なら削除する

## 4. バージョニング（4層に分ける）

### 4.1 HW Rev（機体＋基板の版）

- HW Rev は `r<major>.<minor>` 形式で表す
- `major` はメカを含む機体系世代を表し、基本性能や特性が大きく変わる単位とする
- `minor` は同一機体系世代内の基板版を表し、パターン修正、ピンアサイン変更、IC変更などを表す
- 例:
  - `mini r1.0`
  - `mini r1.1`
  - `classic r1.0`

ルール:
- **メカ構成、寸法、重量配分、駆動系、センサ配置など、基本性能や特性が大きく変わる変更** は `major` を上げる
- **パターン修正、ピンアサイン変更、IC置換など、基本性能が大きく変わらない変更** は `minor` を上げる
- 正式な機体識別名、機体識別ブロック、ログメタ情報では `family + r<major>.<minor>` の対応が取れている状態を保つ
- 機体識別ブロックでは `hw_rev_major` と `hw_rev_minor` を別フィールドで持つ

### 4.2 FW Version（共通ファームの版）

- 共通コード、`platform`、`board`、ログ、通信、NVMインタフェースを含む版
- SemVer（`MAJOR.MINOR.PATCH`）を採用

ルール:
- **MAJOR**: 互換性が壊れる変更
  - モード体系や状態遷移の前提が変わる
  - ログ形式やNVM形式が非互換になる
  - `platform` / `board` / `params` の組み合わせ前提が壊れる
- **MINOR**: 後方互換を保った機能追加・改善
- **PATCH**: バグ修正

### 4.3 Params/Tune Version（機体ごとの調整版）

- 目的は互換性より **再現性**
- `params/<profile>/` ごとに独立して版を管理する

ルール:
- 走りの挙動が変わったら上げる
- 思想が変わるレベルなら大きく上げる
- `params` の意味や単位が変わる場合は FW Version の扱いも合わせて検討する

### 4.4 NVM / Log Schema Version

- 永続化データとログデータは、**各blob/recordが `magic`, `version`, `length`, `crc` を持つ**ことを原則とする
- 機体識別ブロック、校正値、迷路保存、ログヘッダは個別にスキーマ版を持てるようにする

ルール:
- 後方互換がない変更は schema version を上げる
- 読み込み失敗時は既定値やセーフモードへ落とす
- アプリ層から直接 Flash sector 番号や FRAM アドレスを扱わない

### 4.5 `params/<profile>/params.h` の変更ルール（推奨）

- `params/<profile>/params.h` を編集した場合は、PR本文に以下を必ず書く
  - 変更項目
  - 変更理由
  - 実機確認結果（未実施なら未実施と明記）
- 複数機体で共通に持つべきパラメータは、原則として同じ意味と単位を保つ
- 片側だけ差を付ける場合は、理由（機体差、センサ差、モータ差など）を明記する
- `.bak` 等の不要ファイルはコミットしない

## 5. ブランチ運用（実験を前提にしたルール）

### 5.1 基本ブランチ

- `main`
  - いつでもビルドでき、既知の動作を維持するブランチ
  - `main` に入れるPRは、原則として公式サポート中の MCU系列がビルドできること
- `feature/<area>/<topic>`
  - 採用予定の開発
- `fix/<area>/<topic>`
  - バグ修正
- `exp/<area>/<yyyyMMdd>-<short>`
  - 成功するか不明な実験

### 5.2 日常運用ルール

- WIPコミットを許容する
- 15〜30分ごと、大きい変更の前、作業終了前に `commit + push` してバックアップを残す
- コミット粒度は **戻したい単位** を優先する
- 実験ブランチでは、途中で動かなくてもコミットしてよい

### 5.3 実験（exp）の進め方

- 実験は `exp/*` ブランチで行う
- 実験の状態が分かる命名を使う
- 成功した実験は `feature/*` へ整理してから `main` に取り込む
- 失敗した実験は短期的にはブランチを残し、長期的にはタグ化して棚卸しする

例:
- `exp/control/20260404-turn-model-v2`
- `feature/platform/f413-bringup`
- `fix/logging/fram-header-crc`

### 5.4 PR本文に必ず書く項目

- 目的と変更概要
- 影響レイヤ（`common`, `app`, `platform`, `board`, `params`, `nvm`, `stable`, `tools`）
- 対象 MCU系列 / 対象機体
- `.ioc` 変更の有無
- `params` 変更の有無
- NVM / Log schema 変更の有無
- 安定バージョン manifest / runtime settings 変更の有無
- 実機確認の有無

### 5.5 実験の一覧性（推奨）

- `docs/EXPERIMENT_LOG.md` を更新し、何を試したかを残す
- 最低限、以下の情報を残す
  - topic
  - 目的
  - 結果（成功/失敗）
  - 参照（ブランチ名 or タグ名 or コミットID）
  - メモ

### 5.6 ツール類（ホスト側スクリプト等）の運用

- `tools/` の変更は原則として `feature/tools/<topic>` または `fix/tools/<topic>` で行う
- 実験ブランチにツール変更が必要な場合は、可能ならツール側を先に `main` へ取り込んでから追従する
- 原則として、1つのPRに `platform/` と `tools/` の変更を混在させない

推奨:
- CIで「`platform/**` と `tools/**` が同一PRで変更されたら注意を出す」チェックを導入する

## 6. リリースと安定バージョン管理

### 6.1 `main` と通常リリース

- `main` は共通の開発基準線であり、基本動作を維持しつつ機能追加や修正を継続的に取り込む
- `vX.Y.Z` は **mainに入ったものだけ**に付ける
- 実験中のチェックポイントにタグを付けたい場合は、`v` 形式ではなく別プレフィックスを使う
- ビルド識別子はタグとは別にファームへ埋め込む

### 6.2 安定バージョンの定義

- 安定バージョンは、**特定個体で調整済みの機体 + ソフト + 設定の組み合わせ**とする
- `main` の最新版と、各個体の安定バージョンは別物として管理する
- 安定バージョンは `unit_serial` 単位で持つことを原則とする

### 6.3 安定バージョンに固定するもの

- `fw_commit` または `fw_tag`
- `FW_TARGET`
- `family`
- `hw_rev_major`
- `hw_rev_minor`
- `board_id`
- `unit_serial`
- `params/tune version`
- 校正値の版や NVM schema version
- 備考、既知の制約

### 6.4 安定バージョンで後から編集可能にするもの

- `goal_x`, `goal_y` などのゴール座標
- 実行プリセット、既定モード、運用メモ
- これらは `runtime_settings` として安定バージョン本体から分け、**再ビルドなしで編集できる状態**を目指す
- `params` の意味や校正値の前提が変わる場合は、同じ名前のまま更新せず新しい安定バージョンとして切る

### 6.5 管理方法

- 安定バージョンは `stable/<family>/<unit_serial>/<stable_name>/` に置く
- `stable_name` は `sYYYYMMDD-<short>` 形式を推奨する
- 各安定バージョンは少なくとも以下を持つ
  - `manifest.yaml`
  - `runtime_settings.yaml`
  - `notes.md`
- `manifest.yaml` は固定情報、`runtime_settings.yaml` は後から編集可能な運用設定を持つ
- 必要に応じて対応する Git tag `stable/<family>/<unit_serial>/<stable_name>` を付ける
- 書き込みツールは、安定バージョンを選択して **バイナリ + runtime_settings** をまとめて機体へ適用できるようにする

## 7. ビルド識別・機体識別

### 7.1 ビルド識別

- `FW_TARGET`, `FW_BUILD_TYPE`, `FW_BUILD_TIME_UTC`, `FW_GIT_DESCRIBE`, `FW_GIT_SHA`, `FW_GIT_BRANCH`, `FW_GIT_DIRTY` をファームに埋め込む
- 起動時トレースとログヘッダの両方で出せるようにする
- 実機試験結果は、原則としてビルド識別子と対で記録する

### 7.2 共通バイナリの範囲

- **共通バイナリは MCU系列ごとに1個**とする
- `STM32F405` 系向けと `STM32F413` 系向けは別バイナリにする
- 同じ MCU系列の中では、機体識別ブロックにより `board` / `params` / `capability` を切り替える

### 7.3 機体IDの読み出し元

- 機体IDの一次情報は、**MCU内蔵Flashの予約領域に置く機体識別ブロック**から読む
- 予約領域はアプリ書き込み時の通常 erase 範囲から除外する
- 読み出しタイミングは、`HAL_Init()` の後、`board` 依存GPIOや外付けデバイス初期化の前とする

推奨フィールド:
- `magic`
- `schema_version`
- `family`
- `board_id`
- `hw_rev_major`
- `hw_rev_minor`
- `unit_serial`
- `default_param_profile`
- `capability_flags`
- `mcu_uid[3]`
- `crc`

### 7.4 MCU Unique ID の扱い

- `HAL_GetUIDw0()` などで得られる MCU固有UID は、**補助情報**として扱う
- MCU固有UIDを直接 `board_id` として使わない
- 用途は以下に限定する
  - 機体識別ブロック初期化時の紐付け確認
  - ログへの記録
  - 機体識別ブロック破損時の診断

### 7.5 機体識別失敗時の挙動

- `magic`, `version`, `crc` のいずれかが不正なら通常走行に入らない
- セーフモードへ入り、`UID` と異常理由をトレースへ出す
- 必要に応じて診断モードから識別ブロックを書き直す

## 8. トレースとログの方針

### 8.1 2系統に分ける

- **トレース**
  - 目的: mode や case、実行内容、エラー理由などを人が即時に確認する
- **ログ**
  - 目的: 走行・調整用の時系列データを保存し、動作後にまとめて解析する

### 8.2 トレース（操作補助用 `printf` 相当）

- `STM32F413` 系の標準トレース経路は **`SWO`** とする
- 接続は `ST-Link V3 MINIE` を前提とし、`SWDIO`, `SWCLK`, `SWO`, `NRST` を使用する
- 内容は mode/case、開始・終了、設定値、警告、エラー、セーフモード理由などの**低頻度・人間可読文字列**に限定する
- 高頻度ループ内で大量の文字列出力を行わない
- `STM32F405` 系既存機体は現行機体で完了とし、新しい `F405 + FRAM` 派生は作らない
- `STM32F405` 系既存機体は `UART` を維持するが、呼び出し側APIは `trace_printf()` などで抽象化し、メッセージ種別と出力フォーマットは新世代とできるだけ揃える

### 8.3 ログ（デバッグ・調整用）

- `STM32F413` 系の標準ログ保存先は **外付けFRAM** とする
- ログはバイナリで蓄積し、走行や調整用動作の終了後に**専用のダンプモード**でまとめて吐き出す
- ダンプ経路は **`SWO`** を標準とし、PC側ツールで受信・CSV変換・可視化を行う
- 高頻度ログを制御ループ中に文字列化しない
- `STM32F405` 系既存機体は `RAM + UART` の制約内で運用し、ログAPI、ログヘッダ、メタ情報項目は新世代とできるだけ共通化する
- `STM32F405` 系には FRAM 前提の仕様を持ち込まず、backend の容量差だけを吸収する設計にする

### 8.4 ログに必ず含めるメタ情報

- `FW_TARGET`
- `FW_GIT_DESCRIBE`
- `family`
- `hw_rev_major`
- `hw_rev_minor`
- `board_id`
- `unit_serial`
- `params/tune version`
- `mode`
- `case`
- `log_profile`
- `log_schema_version`
- `stable_id`（安定バージョンを選択している場合）

### 8.5 PC側ツールの方針

- `tools/logging/` に、SWO受信、FRAMログダンプ整形、CSV変換、可視化支援をまとめる
- `web_visualizer.py` 相当の表示ツールは、ログメタ情報から機体・ビルドを自動識別できるようにする

## 9. NVM方針

- 永続データ保存は `NVM` という共通概念で扱う
- 実装 backend は `internal flash` と `external FRAM` の両方を許容する
- 用途ごとに保存先を分ける
  - 機体識別ブロック: 内蔵Flash予約領域
  - 校正値 / 設定値 / 迷路: `nvm/` API経由で管理
  - 高頻度ログ: FRAM
- 保存形式は `magic`, `version`, `length`, `crc` を持つ
- アプリ層から生の sector 番号、Flashアドレス、FRAM固定オフセットを直接参照しない

---

## 10. ハードウェア構成の異なる機体への対応

- MCU依存、HAL依存、CubeMX依存の処理は `platform/` に閉じ込める
- ピン配置、搭載部品、極性、搭載能力は `board/` で定義する
- 探索、経路導出、制御の本体は `common/` / `app/` に置く
- 条件付きコンパイルは `platform/` / `board/` 境界で閉じることを原則とし、共通アルゴリズム層に `#ifdef` を拡散させない

---

## 11. 機能フラグ管理の方針

- 状態、機能有効化、機体能力、デバッグ用無効化を分けて扱う
- 命名規則:
  - `status_*`: 現在状態
  - `enable_*`: 機能有効化
  - `has_*`: 機体能力
  - `disable_*`: デバッグ用一時無効化
- デバッグ用無効化フラグは1か所に集約し、通常機能の設定と混ぜない
- 高頻度制御ロジックに場当たり的なグローバルboolを増やさない

---

## 12. 教育用基本機能の切り分け

- 競技用ファームをそのまま配布しない
- `competition_full`, `education_basic`, `education_intermediate` のような機能段階を定義できる構造にする
- 教育用ビルドでは、高度な補正や競技向け最適化を明示的に無効化できるようにする
- 基本機能と高度機能はモジュール境界を分けて管理する

---

## 13. 決め（今回合意した内容）

- 機体の大別は **`mini` / `classic`** を維持し、正式な機体識別名は **`<family>_r<major>_<minor>`** とする
- HW Rev は **`major = 機体系世代`、`minor = 基板版`** とする
- 共通バイナリの範囲は **MCU系列ごとに1個**とする
- 機体IDの一次情報は **MCU内蔵Flash予約領域の機体識別ブロック**から読む
- `STM32F413` 系の操作補助トレースは **`SWO`** を標準とする
- `STM32F413` 系のデバッグ・調整ログは **`FRAM` に保存し、走行後に `SWO` からダンプ**する
- 安定バージョンは **個体ごとに調整済みの組み合わせ**として `main` とは別に管理する
- `STM32F405` 系既存機体は現行機体で完了とし、`RAM + UART` の範囲で新世代とできるだけ共通のトレース/ログAPIを使う
- `MCU Unique ID` は補助情報であり、機体切り替えの主キーにはしない
- `_sandbox/` の成果物は正式ディレクトリへ移してから運用する
